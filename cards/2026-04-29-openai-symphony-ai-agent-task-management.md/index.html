OpenAI 开源 Symphony：给每一个任务配一个永不下班的 AI 员工

OpenAI 开源 Symphony：给每一个任务配一个永不下班的 AI 员工

来源： https://x.com/vista8/status/2049484504444834126 日期： 2026-04-29 作者： 向阳乔木 (@vista8) 项目链接： https://github.com/openai/symphony Star 数： 25.7k

项目概述

OpenAI 开源了 Symphony 项目，这是一个给 AI Agent 用的任务管理系统。OpenAI 内部与 Linear 整合，大大提升了人管理 Agent 的能力。

核心概念

Symphony 将项目工作转化为隔离的、自主的实现运行，让团队能够管理工作而不是监督编码 Agent。

演示场景

Symphony 监控 Linear 看板上的工作，生成 Agent 来处理任务。Agent 完成任务并提供工作证明：

  • CI 状态
  • PR 审查反馈
  • 复杂度分析
  • 演练视频

当被接受时，Agent 安全地提交 PR。工程师不需要监督 Codex；他们可以在更高层次上管理工作。

技术规格

系统架构

Symphony 是一个长时间运行的自动化服务，持续从问题跟踪器（当前版本为 Linear）读取工作，为每个问题创建隔离的工作区，并在工作区内运行编码 Agent 会话。

解决的四个运营问题

1. 将问题执行转变为可重复的守护进程工作流，而不是手动脚本 2. 隔离 Agent 执行在每问题工作区中，Agent 命令只在每问题工作区目录内运行 3. 将工作流策略保留在仓库中（WORKFLOW.md），团队将 Agent 提示和运行时设置与代码一起版本化 4. 提供足够的可观察性来操作和调试多个并发 Agent 运行

主要组件

1. Workflow Loader - 读取 WORKFLOW.md，解析 YAML front matter 和提示正文 2. Config Layer - 暴露工作流配置值的类型化 getter 3. Issue Tracker Client - 从问题跟踪器获取候选问题 4. Orchestrator - 拥有轮询周期和内存运行时状态 5. Workspace Manager - 映射问题标识符到工作区路径 6. Agent Runner - 创建工作区，构建提示，启动编码 Agent 7. Status Surface（可选）- 呈现人类可读的运行时状态 8. Logging - 发出结构化运行时日志

核心配置


tracker:
  kind: linear
  endpoint: https://api.linear.app/graphql
  api_key: $LINEAR_API_KEY
  project_slug: your-project
  active_states: [Todo, In Progress]
  terminal_states: [Closed, Cancelled, Done]

polling:
  interval_ms: 30000

workspace:
  root: /tmp/symphony_workspaces

hooks:
  after_create: |
    # 工作区创建后运行
  before_run: |
    # 每次 Agent 尝试前运行
  after_run: |
    # 每次 Agent 尝试后运行

agent:
  max_concurrent_agents: 10
  max_turns: 20
  max_retry_backoff_ms: 300000

codex:
  command: codex app-server
  turn_timeout_ms: 3600000
  stall_timeout_ms: 300000

关键特性

1. 隔离工作区 - 每个问题有独立的工作区目录 2. 并发控制 - 全局和每状态并发限制 3. 重试机制 - 指数退避重试，最大 5 分钟 4. 动态重载 - WORKFLOW.md 更改自动检测和应用，无需重启 5. 可观察性 - 结构化日志，可选的 HTTP API 和仪表板 6. 安全 - 工作区路径验证，清理标识符，秘密处理

工作流

1. 轮询 Linear 获取活跃状态的问题 2. 为每个问题创建隔离工作区 3. 从 WORKFLOW.md 模板构建提示 4. 启动 Codex Agent 处理问题 5. Agent 完成工作，提供证明 6. 工程师审查并接受 PR 7. Agent 安全提交代码

安全考虑

  • 工作区路径必须在配置的工作区根目录下
  • Agent 工作目录必须是每问题工作区路径
  • 工作区目录名必须使用清理的标识符
  • Hook 脚本是完全受信任的配置
  • 实现应明确评估风险配置文件并加固执行工具

关键洞察

1. 从监督到管理 - Symphony 代表了从"监督编码 Agent"到"管理工作"的范式转变 2. 工作流即代码 - WORKFLOW.md 将 Agent 提示和运行时设置与代码一起版本化 3. 隔离执行 - 每个问题在独立工作区中运行，防止交叉污染 4. 可观察性优先 - 结构化日志和可选 API 让团队能够调试和优化 Agent 行为 5. 渐进式采用 - 规范是语言无关的，团队可以用任何语言实现
