Loop Engineering 到底是什么？看这一篇就够了

Author: Henry Li（DeerFlow 团队） Source: https://mp.weixin.qq.com/s/CdSaX5eiAvhcYkNj4qZ6Nw Publisher: TRAE.ai Date: 2026-06-26 Type: WeChat MP Article (Long-form) Extracted via: Browser extraction

Why（背景动机）

过去几个月，很多人都在反复做同一件事：写 Prompt → 交给 Coding Agent → 等结果 → 判断哪里不对 → 再写 Prompt。时间一长会发现：这本身就是一个循环，只不过循环体的执行者是人自己。

Agent 干完一轮就停下，安静等着。真正负责"判断、决定、再来一轮"的角色由人扮演。人，被绑在了循环里最频繁的那个位置上。

代价：

  • Agent 每跑完一轮都得回到屏幕前查看，否则循环就停在那儿不动
  • 注意力被反复切走又切回，一天下来注意力被切得很碎
  • 社区里甚至出现过用软件通知、ESP32 硬件来提醒"Agent 已经跑完了"的小工具

核心问题：既然其余环节 Agent 都能跑，那个"再来一轮"的位置，是不是也能交给 Agent？这正是 Loop Engineering 要回答的问题。

What（核心定义）

Loop Engineering 的本质

一句话：把 Loop 的执行者从"你"换成"Agent"。

原本由你亲自跑的"发现问题—想办法—动手—检查"循环，现在交给 Agent 去跑。你要做的，只剩开头把目标（验收标准）说清楚。

Closed Loop 的完整结构

1. 目标：由 human 通过 /goal 设定一次。方向得人来定，这是唯一不该交出去的事 2. Discovery：Agent 自己去发现这一轮要做什么 3. Plan：把发现的事拆成清晰、可执行的步骤 4. Execute：动手做。可采取 fan-out 模式，多个 Agent 并行 5. Verify：检查目标有没有达成。这是 Loop Engineering 里最重要的一环 6. Memory：进度记录在对话之外，独立于任何一次对话而存在 7. Ship 或 Iterate：达标就 Ship，不达标就带着问题回到 Discovery

关键概念

Self-prompting：上一轮跑完后，让 Agent 根据已有进展自己写下一轮要执行的 Prompt。人不再负责接力，循环自己续上了。

Memory 在对话之外：把进度写进文件、写进结构化的状态记录。下一轮无论由哪个 Agent 来跑，都能读到"现在到哪儿了"。Memory 不在对话里，循环才稳得住。

Orchestrator（编排）：掌管目标、负责派活，等各路 Agent 干完后汇总结果，再决定是 ship 还是 iterate。

Fan-out / Fan-in：

  • Fan-out：一个 Orchestrator 把任务拆开，同时派给多个 Agent 并行去做
  • Fan-in：这些 Agent 各自跑完后，结果再汇聚回来，由一处统一收口

Open Loop vs Closed Loop

维度 Open Loop Closed Loop
入口 一句话"去看看该做什么" 有界目标 + 明确验收
探索面 广，常发现意想不到的方向 窄，基本不会给意外惊喜
开销 难预估，靠外部预算上限兜住 可控，能估出"大概几轮收敛"
适用场景 预算充裕、以探索为目的 交付场景，在可预期开销内做完
建议 不要一上来就 Open Loop 多数情况先从 Closed Loop 开始

Crewlet 案例：Open Loop 管理创业公司

Crewlet 的 Agent 正在管理一个名为 SuperDesign 的创业公司，把以下数据源全部交给 OpenClaw Agent：

  • GitHub 代码仓库、Sentry 日志
  • Google Analytics 埋点与行为数据
  • Supabase 业务主库、AB 实验数据
  • Stripe 支付、X / Discord 社交数据、SEO、GEO

Agent 每隔一段时间"醒来"，自主查看核心经营数据与近期错误日志，能做的事包括：

  • 发现经营指标问题并归因
  • 从错误日志中定位 Bug，直接改 GitHub 代码并提交 PR 上线
  • 发现大量用户卡在付费这一步，给即将到期的用户发送挽回邮件
  • 改 Agent 代码、增强 Prompt Caching，提性能、省开支
  • 发起 AB 实验并分析报告，进而迭代自身功能

How（具体方法/参数）

Closed Loop 实践：为工具库补测试

验收标准（客观可判定）： 1. tests/acceptance/ 下的所有用例通过 2. npm run typecheck 无报错 3. 现有测试不出现新的失败

循环 Prompt 骨架： 


/goal 目标：为 <工具库> 补齐测试并修复其中的 bug。

验收标准（全部满足才算完成）：
- tests/acceptance/ 下的所有用例通过
- `npm run typecheck` 无报错
- 现有测试不出现新的失败

循环规则：
- 跑 `npm test`。
- 若未全绿：读取失败用例与报错，定位并修复，然后重跑。
- 只要不全绿，就重复上一步，直到全绿再停。
- 中途若验收标准本身有歧义，先提问，不要自行猜测。

Closed Loop 为什么能成立：

  • 验收是客观的（测试、typecheck、benchmark 给出 0/1 信号）
  • Agent 不需要去猜"我做完了没"，判断被外包给了一套确定性检查
  • 循环有了必须收敛的客观靶子，轮数和开销也就有了大致上界
  • 把模糊的"做好"换成可判定的"全绿"，Closed Loop 就立住了

Closed Loop 实践：网店做增长

Orchestrator 派出三个 Sub-Agent： 1. Builder：构建 lead magnet（引流磁铁） 2. Scout：选题排序 3. Growth：落地动作

汇总与记忆：

  • 把三个 Sub-Agent 的产出合并成统一的行动计划
  • 写回 next_steps——这份文件就是这套循环的跨周期记忆
  • 下一轮开工，Orchestrator 第一件事就是再读它

循环条件（人为设定的边界）：

  • 站点这周是否还在增长？
  • Scout 手里是否还压着 ≥3 条未消化的新选题？
  • 下一个 lead magnet 是否已经定下来？
  • 条件没满足就接着转；都满足了就暂停

通用 Loop 骨架

抽干净后，剩下的就是这条线：

定时触发 → 去读点什么 → 打分/过滤 → 产出草稿 → 比对历史去重 → 必要时让人过目 → 记录下来 → 下一周期再从头转

四个速写场景

1. 自由职业者：每周五扫项目文件夹，为客户起草个性化进度更新，草稿进 review 文件夹等过目 2. 学生/追新的人：每周日搜领域进展，按相关度打分过滤，写成通俗简报 3. 网店老板：每月 1 号读销售数据，重写低转化商品描述，记录改了什么为什么改 4. 创作者：每周一读选题清单 + 90 天表现数据，给选题打分排序 top 5

So What（启示与行动项）

对行业的启示

1. Loop Engineering = 数字员工的开始

  • 从"人当循环体"到"Agent 当循环体"是质的飞跃
  • 不是"让 Agent 自动跑起来"，而是"搭一个能替你转动的机制"
  • 最容易被忽略的一点：它得懂得在恰当的时候停手，把决定权重新交回到人手上

2. Closed Loop 是生产级 Agent 的起点

  • 把模糊的"做好"换成可判定的"全绿"
  • 验收标准客观 → 循环可收敛 → 开销可预估
  • 多数情况先从 Closed Loop 开始，跑顺了再考虑 Open Loop

3. Memory 在对话之外是核心创新

  • 不是把所有进展堆回 Context Window
  • 而是写进文件、写进结构化的状态记录
  • 让循环独立于任何一次对话而存在

对 A 哥的启示

1. Multica 团队应该按 Loop Engineering 模式设计

  • 每个角色 = 一个 Closed Loop（目标 + 验证器 + Memory）
  • Orchestrator 掌管目标，Fan-out 给 Sub-Agent 并行
  • 验证器是 Loop 的核心（因果链 #15 的实战验证）

2. FDE 的核心能力是设计 Loop

  • 帮客户把"手工循环"升级为"Agent 循环"
  • 关键不是"用哪个模型"，而是"验收标准怎么定"
  • 把模糊的"做好"换成可判定的"全绿"

3. 创业机会：Loop Engineering as a Service

  • 帮企业设计 Closed Loop（研发、运营、增长）
  • 参考 Crewlet 案例：用 Open Loop 管理整个创业公司
  • 目标客户：还在"人当循环体"的传统团队

关键洞察

Loop Engineering 的本质不是"让 Agent 自动跑起来"，而是"把循环体的执行者从人换成 Agent"。你从循环体里退出来，退回到设目标、定边界的位置。变的不是循环本身，是谁来当那个一刻不能离开的执行者。

创业机会

1. Loop Engineering 咨询：帮企业从"手工循环"升级到"Agent 循环" 2. Closed Loop 模板库：为不同场景（研发/运营/增长）提供预构建的 Loop 模板 3. Memory 基础设施：为 Agent 循环提供跨周期记忆服务
