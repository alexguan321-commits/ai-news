I Made My Hermes Agent 10x Faster Without Changing the Model

wandermist (@wandermist) 2026-06-30 Hermes Agent | 文件夹结构 | INDEX.md | Agent优化 | 工作流 👁 - views

Why（背景动机）

作者使用 Hermes Agent 时发现一个荒谬的对比：AI 能设计对抗未知冠状病毒的疫苗（Cambridge 已进行人体试验），但当让 Agent 从文档列表中获取最新简报时，它要花近两分钟打开错误的文件，最后给出三个月前的简报。

核心问题：Agent 的能力足够强，但文件夹结构成了瓶颈。Agent 在导航上浪费了大量能力，而不是做实际工作。

关键洞察：围绕 Agent 的隐形脚手架（invisible scaffolding）比 Agent 自身的能力更重要。

What（核心定义）

问题诊断

作者追踪了 5 个常见任务在重构前的表现：

任务 打开文件数 耗时
找到当前文章简报 7 2:00
找到品牌颜色定义 5 1:12
查找文章队列用于规划 4 0:48
找到同一主题的过往文章 6 1:36
提取发布推广策略 3 0:34

根本原因：文件夹按内容类型组织（Articles、Research、Assets、Strategy），这是人类思维的方式。但 Agent 不按类别思考——当要求规划产品发布时，任务需要从分散在不同文件夹的策略笔记、品牌指南和过往发布中提取。

Agent 必须每次都搜索所有地方，不知道哪个草稿是当前的、哪个是归档的。

解决方案：INDEX.md 模式

三个规则协同工作：

1. 每个主要文件夹根目录的 INDEX.md 是地图

  • 列出每个子文件夹和规范文件
  • 告诉 Agent 应该从哪里开始
  • 相当于软性审批门控：Agent 在知道自己在处理什么之前不允许开始工作

2. 按关注点（concern）组织，而非内容类型

  • 每个关注点有自己的文件夹
  • 品牌工作留在品牌文件夹，策略留在策略文件夹
  • Agent 不需要跨越边界寻找不属于那里的东西

3. 文件夹编号明确阅读顺序

  • `01.Brand System` 在 `02.Editorial Strategy` 之前读取
  • Agent 不猜测
  • 文件夹内的文件也编号，Agent 知道 `01.Articles` 是起点

重构后的性能

任务 打开文件数 耗时 改进
找到当前文章简报 1 0:10 12x
找到品牌颜色定义 3 0:22 3.3x
查找文章队列用于规划 1 0:10 4.8x
找到同一主题的过往文章 2 0:18 5.3x
提取发布推广策略 1 0:12 2.8x

最慢任务从 2 分钟降到 26 秒，最快任务约 10 秒。

How（具体方法/参数）

INDEX.md 的三次迭代

1. 第一版：列出品牌文件夹内的每个文件，40 行

  • 技术上完整，但 Agent 每次都要解析，长度本身就是开销

2. 第二版：太短，约 15 行

  • Agent 仍然问没回答的问题

3. 第三版（最终版）：只列出子文件夹和规范文件

  • 包含简短的 "Where To Go" 部分，告诉 Agent 起点

文件夹结构示例


05.Brand/
├── INDEX.md                    # 地图
├── 01.Brand System/
│   ├── 01.Brand Guidelines.md
│   └── 02.Color Palette.md
├── 02.Editorial Strategy/
├── 03.Promotion/
│   └── 01.Promotion Strategy.md
├── 04.Previous Articles/
└── 06.Archived/                # 旧简报和退役计划

关键设计：

  • 归档内容放在 `06.Archived`，Agent 默认不去那里
  • 除非特别要求历史上下文
  • 这种分离保持活动文件夹干净、搜索快速

五个常见错误

1. 在每个子文件夹添加 INDEX.md

  • 太多地图意味着 Agent 花时间读索引而不是工作
  • 只在主要文件夹根目录保留 INDEX.md

2. 编号必须完美

  • 编号不需要完美，只要方向性正确
  • 完美排序的文件夹不是目标

3. 在 Agent 显示困惑之前构建结构

  • 大多数人过度工程化 Agent 设置
  • 原则：使用最小的接口完成工作
  • 只在 Agent 迷路时添加结构，只添加足够修复该特定问题的结构

4. 在结构内嵌套结构

  • 把两级层次变成五级
  • Agent 必须读取多个 INDEX 文件并解析多个编号序列
  • 深度是快速文件查找的敌人

5. 重命名文件夹但不更新 INDEX.md

  • 对旧名称的引用会中断
  • 一旦设定文件夹名称就保持不变
  • 稍微尴尬的名称比损坏的引用更便宜

诊断测试（5 分钟）

在重组任何文件夹之前，对 Agent 最常处理的 3 个真实任务运行此测试：

1. 计时每个任务 2. 记录它在找到正确文件之前打开多少文件 3. 追踪它多频繁选择错误文件或停下来问你应该用哪个

判断标准：

  • 任何任务超过 30 秒或打开 3 个以上错误文件 → 该任务背后的文件夹坏了
  • 选择最常发生的失败
  • 打开该文件夹，写一个 INDEX.md
  • 重新运行同一任务
  • 如果 Agent 在 30 秒内找到正确文件 → 修复成功

局限性

1. 归档内容有时仍被引用

  • Agent 必须知道去 `06.Archived` 而不是活动文件夹
  • 在 INDEX.md 中提到这一点

2. Obsidian Sync 跨设备偶尔不同步

  • 如果在笔记本上更新文件但旧版本仍在 VPS 上，Hermes 加载 VPS 副本
  • 这是同步问题而非文件夹结构问题

3. 编号在文件夹过多时失效

  • 超过 10-12 个编号项目，数字变得任意
  • 保持主要类别低于该阈值

4. 对纯代码或数据密集型设置可能不适用

  • 这种结构最适合内容密集和工作流密集的场景

So What（启示和行动项）

对 AI 工程师的核心启示

1. 围绕 Agent 的脚手架比 Agent 自身能力更重要

  • 能力在脚手架损坏时很便宜（浪费）
  • 构建脚手架，能力自己会处理

2. 按关注点组织，而非内容类型

  • 人类按类别思考，Agent 必须手动交叉引用
  • 按任务/项目/关注点组织文件夹

3. INDEX.md 是最小但最有效的修复

  • 一个索引文件 + 文件夹名前的几个数字
  • 就是迷路的 Agent 和工作的 Agent 之间的区别

4. 深度是快速文件查找的敌人

  • 大多数重组增加深度以为那是精确
  • 保持层次扁平

5. 先测量，再重组

  • 不要过度工程化
  • 只在 Agent 迷路时添加结构

对 A 哥当前系统的映射

文章概念 A 哥当前实现 差距/改进
INDEX.md 模式 未使用 可立即应用
按关注点组织 部分（AI_News 按类型，LLM Wiki 按主题） 可优化
文件夹编号 未使用 可为主要目录添加
归档分离 未明确 应添加 Archived/ 目录
诊断测试 未系统测量 应运行 5 分钟测试

立即可执行

1. 本周：运行 5 分钟诊断测试

  • 选择 Hermes 最常处理的 3 个任务
  • 计时并记录打开文件数
  • 识别最失败的文件夹

2. 本月：为主要目录添加 INDEX.md

  • `~/AI_News/` — 资讯采集系统
  • `~/LLM_WIKI/` — 知识库
  • `~/projects/` — 项目目录

3. 持续：遵循"最小接口"原则

  • 只在 Agent 迷路时添加结构
  • 只添加足够修复该特定问题的结构

对 FDE 团队的启示

  • Agent 的脚手架比 Agent 本身更重要
  • 团队的知识管理结构决定了 AI 助手的效率
  • 混乱的文件夹 = 浪费的 AI 能力
  • INDEX.md 是团队知识地图的最小实现
  • 每个项目/领域一个 INDEX.md
  • 告诉 Agent 从哪里开始
  • 按关注点组织，而非内容类型
  • 项目 A 的所有文档在一个文件夹
  • 而不是所有"策略"文档在一个文件夹

关键洞察

"Capability is cheap when the scaffolding around it is broken. Build the scaffolding, and the capability takes care of itself."

当脚手架损坏时，能力很便宜（被浪费）。构建脚手架，能力自己会处理。一个索引文件和文件夹名前的几个数字，就是迷路的 Agent 和工作的 Agent 之间的区别。
