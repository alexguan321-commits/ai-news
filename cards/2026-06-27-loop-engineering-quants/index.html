Loop Engineering for Quants: System That Researches, Verifies and Trades While You Sleep

Author: Atlas (@crptAtlas) Source: https://x.com/crptAtlas/article/2070573863218368823 Date: 2026-06-27 Type: X Article (Long-form) Extracted via: Chrome AppleScript

I built a quant system that pulls market data, researches alpha, checks its own signals and places trades on its own. I have not typed a prompt into it in weeks. It still runs.

It pulls data on a schedule. It runs alpha research. It verifies every signal through a separate agent. It only executes what passes. It writes every lesson it learns back to memory so the next run is a little sharper.

This is everything I figured out about loop engineering and how to wire it into a complete autonomous trading system. No fluff, just the build.

By the end you'll know 3 things:

  • The real difference between prompting an agent and engineering a loop
  • The 6 pieces that run every working loop in production
  • How to wire those 6 pieces into a self-improving quant system from scratch

Part 1: Prompting vs Loop Engineering

For the last 2 years, working with AI looked like one thing. You typed a prompt, you read what came back, you typed the next prompt based on what you saw. You were the loop.

The agent was a tool and you held it the whole time. Every move was you sitting at the keyboard deciding what happens next.

Loop engineering ends that. You stop being the thing inside the loop and become the architect who designs it.

A loop is a recursive goal. You define a purpose, the agent iterates against it, and the loop keeps running until a real stopping condition is met. The agent forgets between runs. The loop does not. That single fact is the whole architecture.

Boris Cherny, who runs Claude Code at Anthropic, put it plainly: "I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops." He stopped typing instructions one at a time and built systems that send the instructions for him, read the results and decide what happens next.

For coding, that changes how software ships. For trading, it changes everything.

No quant ever got rich by typing one prompt and walking away. The edge comes from running the same cycle thousands of times, getting one percent better every iteration and never sleeping. That is exactly what a loop does.

A lot of retail quants will read this and assume it doesn't apply to them because they're small. The opposite is true. The smaller your capital, the more this matters, because a self-running loop is the only way a solo builder ever closes the gap with a fund running 100 PhDs.

Quant trading is already a loop. Pull data, generate signals, backtest, execute, monitor risk, repeat. Every fund on Wall Street runs that exact cycle. Renaissance has been running it since 1988. Citadel runs it with teams of engineers watching every stage. Two Sigma, Jane Street, all of them.

The only real difference is they need hundreds of humans sitting inside the loop. You don't.

If you're still typing prompts into Claude one trade at a time, the leverage point has already moved a floor up. You are not writing better prompts anymore. You are writing the system that writes the prompts.

Part 2: The 6 Pieces That Run Every Working Loop

A working loop is built from 6 parts. Miss one and the loop breaks quietly.

1. The automation

This is the heartbeat. A cron schedule, a webhook, a /loop command or a hook inside Claude Code that fires without you typing.

Two flavors are worth knowing:

  • /loop — reruns on a cadence no matter the state
  • /goal — keeps going until a condition you wrote is actually true, with a separate small model grading whether the work is done

In trading, /loop is your data pull every minute. /goal is "keep iterating on this signal until the backtest Sharpe is above 1.5."

2. The skill

A skill is a procedure manual the agent reads instead of being told everything from scratch each session. It lives in a SKILL.md file and holds your conventions, your rules and your "we don't do it like this because of that one incident."

Without skills, every loop run starts from zero. With skills, intent compounds.

3. The state file

A markdown file, usually called STATE.md or PROGRESS.md. It survives between runs. The agent forgets, the file does not. The agent reads it at the start of every run and writes back what happened at the end.

It sounds too dumb to matter. It is the spine of every working loop.

4. The verifier

The agent that wrote the code is the worst possible judge of whether the code is correct. The same goes for trading. The agent that generated the signal is the worst possible judge of whether that signal is real alpha or noise.

You need a separate agent, with different instructions and ideally a different model, whose only job is to verify the work.

This is the maker-checker pattern, and every prop shop on Wall Street is built this way. At Jane Street, the trader who proposes a trade does not approve it. At Citadel, the researcher who builds the model does not validate it.

5. The worktrees

The moment you run more than one agent against the same files, they start colliding. Git worktrees give each agent its own isolated working directory pointed at its own branch.

In trading, this lets you run signal research, backtesting and risk monitoring in parallel without ever stepping on each other.

6. The connectors

A loop that can only read files is a tiny loop. Connectors built on the Model Context Protocol let it hit a broker API, query a database, post to Slack or send orders to the exchange.

This is the difference between a loop that suggests trades and a loop that actually places them.

These 6 pieces are universal. They show up in Claude Code, in Codex and in every working agentic system on the planet.

Part 3: Wiring It Into a Self-Improving Quant Loop

The quant trading loop has 5 stages. Each stage is its own sub-loop, with its own skill, its own state file and its own verifier.

Stage 1: Data ingestion

An automation fires every minute, every hour or every day depending on the asset class. 


@loop(interval="1h")
def ingest_data():
    data = fetch_market_data(symbols=universe, lookback="30d")
    state.write("latest_data.parquet", data)

The data lands in a shared state file the next stage reads.

Stage 2: Signal generation

This is where the alpha research happens. 


@loop(trigger="data_updated")
def generate_signal():
    data = state.read("latest_data.parquet")
    signal = claude.run_skill("alpha_research", data)
    state.write("pending_signal.json", signal)

The signal agent reads from a SKILL.md file that holds your alpha research rules.

alpha_research_skill.md: 


Goal
Generate signals using linear regression on the last 30 days
of price and volume data.

Rules
- Sharpe ratio must be above 1.5 in 3 of the last 5 backtests
- Position size capped at 2 percent of capital per signal
- Skip signals on FOMC announcement days
- Skip signals 48 hours before earnings releases

Lessons learned
- 2026-02-14: lost 4.2 percent during earnings week.
  New rule: skip any signal 48 hours before earnings.
- 2026-03-08: sector exposure breach caused a 6 percent drawdown.
  New rule: cap sector exposure at 30 percent.
- 2026-04-22: momentum signal blew up on an FOMC day.
  New rule: kill all momentum signals on FOMC days.

The skill grows over time. Every loss writes a new lesson back. Every lesson becomes a rule for the next run. That is what makes the system self-improving.

Stage 3: Verification

The signal goes to a completely separate agent. Different model, different instructions, no exposure to how the original signal was reasoned. 


@checker
def verify_signal(signal):
    result = claude.invoke(
        skill="backtest_verification_skill.md",
        signal=signal,
        rules=[
            "Sharpe ratio above 1.5",
            "Max drawdown below 10 percent",
            "Newey-West t-stat above 2.0",
            "Out of sample period at least 2 years"
        ]
    )
    return result.verdict

If verification fails, the signal is killed. If it passes, it moves to execution. The verifier never sees what the maker reasoned, and that separation is the whole edge.

You can also use a stronger model for the checker than the maker. Opus for verification, Sonnet for generation. Different architectures catch different kinds of errors, which is the same logic ensemble methods use in machine learning.

Stage 4: Execution

Only verified signals reach this stage. 


@auto_mode
def execute(signal):
    if verify_signal(signal):
        broker.send_orders(signal, max_position=0.02)
        state.write("active_trades", signal)

The MCP connector handles the broker API. The loop never asks for permission. Auto mode lets it run hands-off.

Stage 5: Risk monitoring

Running in a parallel worktree the entire time. 


@loop(interval="1m")
def monitor_risk():
    positions = broker.get_positions()
    if drawdown(positions) > 0.05:
        broker.close_all()
        state.append("STATE.md", "Drawdown trigger hit. All positions closed.")

This is the kill switch. It enforces the rules with no negotiation.

Together, these 5 sub-loops form one self-running system. Data flows in, signals get generated, signals get verified, verified signals get executed, risk gets monitored and lessons get written back to memory. Then it starts over.

I designed this once. I have not prompted any of these steps since.

One warning. A loop with no real stopping condition fails quietly. The agent emits a completion signal believing a half-done job is finished, the loop exits and the bad trade sits open. Your stop conditions need to be checkable by something other than the agent's own claim. "Sharpe above 1.5 over the last 30 trades." "Drawdown below 5 percent." "Test suite passes." Never "the agent says it's done."

The Takeaway

Quant trading is already a loop, and every fund on Wall Street runs it. They just need hundreds of humans sitting inside. Loop engineering removes the humans.

6 pieces make every working loop:

  • Automations fire the heartbeat
  • Skills hold the project knowledge
  • State files hold the memory
  • Verifiers grade the output
  • Worktrees isolate parallel work
  • Connectors give the loop hands in the real world

Wire them around the 5-stage trading cycle and you have a self-improving system that runs alpha research, verifies signals, executes trades and monitors risk on its own.

It gets smarter every cycle. Every loss writes a lesson. Every lesson becomes a rule. After 100 trades the skill file is a living document. After 1,000 it is closer to institutional knowledge than anything a single person could hold in their head.

The leverage moved up a floor. You are no longer the trader typing one idea at a time. You are the architect who designed the loop that trades for you while you sleep.
