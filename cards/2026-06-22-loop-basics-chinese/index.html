LOOP 基础使用方法

Author: Easycompany (@Easycompany333) Source: https://x.com/Easycompany333/article/2069019238283849954 Date: 2026-06-22 Extracted: 2026-06-29 Tags: loop-engineering, chinese, tutorial, basics, cost-analysis

AI 已经被大家用了好几年，但大多数每天使用 AI 的人，仍然在用最慢的方式：

输入一个请求，等待回答，自己修改，再继续追问，问题并不在于更快的方法很复杂，真正的问题是：很少有人把更快的方法讲清楚。

更快的方法叫 Loop。

现在，全世界最厉害的一批 AI 工程师，真正关心的就是这件事，这篇文章想补上的，就是很多人一直没讲明白的那部分，读完以后，你会比时间线上大多数人都更懂 Loop：

它到底是什么，底层怎么运转，什么时候值得做，什么时候会变成坑，怎么在 Claude 或 ChatGPT 里搭一个基础版本，以及哪些轻量 Loop 可以直接用在生活里。

大多数人是怎么用 AI 的？

先看最常见的使用方式：

一次只问一个问题，你来决定问什么、你来判断答案好不好，你来决定下一步怎么问，如果你不推动，AI 就不动，如果你停下来，AI 也停下来。

这种方式当然可以用，但上限很明显，你才是发动机，AI 只是你手里的工具，工具本身不会主动工作。

还有另一种方式，这也是现在很多顶级工程师正在转向的工作方式：

你不再一步一步带着 AI 走，你只给它一个目标，然后让它自己跑完整个过程。

它会规划 它会执行 它会检查结果 它会修复薄弱的地方。

然后继续重复，直到目标完成，哪怕人为不直接参与问答与工作方向，工作依然继续往前跑。

Peter Steinberger 说过一句话：

你不应该再给 coding agents 写提示词了。 你应该设计能提示 agents 的 loop。

很多人看到这种话，会觉得很有道理，但心里其实并不知道怎么落地。

所以我们拆开讲

什么是 Loop？

Prompt 是一条单次指令，Loop 是一个持续推进的目标，你可以把它理解成一个循序渐进的任务：

你定义一个目的，AI 围绕这个目的不断迭代，直到完成，Prompt 给你一个答案，然后等你决定下一步，Loop 会自己跑完整个循环： 


DISCOVER  → 找出需要做什么
PLAN      → 决定怎么做
EXECUTE   → 执行任务
VERIFY    → 对照目标检查结果
ITERATE   → 还没达到目标，就把结果喂回去继续

这五步里，真正容易出问题的是三个部分。

1. Verify：验证是 Loop 的心脏

没有真正的结果检查，就没有 Loop，你得到的只是一个 agent 在反复赞同自己，验证机制，才让重复变成进步

它可以是硬测试：代码测试是否通过？ 也可以是可测量条件：数字是否超过 X？ 也可以是评分规则：模型按照 rubric 给结果打分。

如果没有关卡，agent 就是在批改自己的作业，而完成任务的那个模型，通常会对自己太宽容。

2. State：状态让 Loop 学会避免重复犯错

每跑一轮，AI 都必须记住自己试过什么，否则它会永远重复同一个错误，真正的 Loop 会在旁边保留一个小记录：

什么已经完成？ 什么失败了？ 下一步做什么？

这样明天再跑的时候，它可以接着做，而不是从零开始，不过，这也是成本开始变高的地方，后面会讲。

3. Stop condition：停止条件让 Loop 保持理智

没有退出条件的 Loop，会一直跑到成功、崩掉，或者把你的钱花完，严肃的 Loop 一定有两个出口：

成功，停止。 达到硬性上限，停止并汇报。

比如： 最多尝试 8 次。 8 次还没通过，就停下来告诉我哪里失败了，跳过这一步，你就造出了一台可以整晚空转的机器，所以，Prompt 是给 AI 一条指令，Loop 是给 AI 一个任务、一个完成标准，以及一个放弃规则。

你真的需要 Loop 吗？

很多文章一上来就推销 Loop，但不会告诉你什么时候不该用，真正认真做这件事的人，会先看四个条件。

只有这四个都满足，Loop 才值得搭：

1. 任务至少每周重复一次

如果一个任务只是偶尔做一次，搭 Loop 的成本很难回本，一次性任务，用一个好 prompt 更合适。

2. 坏结果可以被自动拒绝

比如： 测试失败 类型检查失败 构建失败 lint 报错 不符合硬规则

如果没有东西能替你判定失败，Loop 很容易空转。

3. Agent 能从头到尾完成任务

如果它做一半就要把大量工作丢回给你，这就不是完整 Loop。

4. "完成"必须客观

如果结果好坏主要看审美和主观判断，人类依然要在关键位置把关。

缺任何一条，就先别做重型 Loop。

说实话：

Loop engineering 确实是个真实方向，但大多数人暂时还不需要很重的版本，真正适合普通人的，是轻量版本，不过你要先知道边界在哪里。

为什么 Loop 最先在代码里爆发？

Loop 最早在软件工程里流行起来，因为代码最容易验证，测试通过就是通过，测试失败就是失败，没有太多争论空间，所以 AI 很容易知道自己有没有完成任务。

一个 coding loop 通常会长这样： 


LOOP SPEC

GOAL:
让 /tests/auth 里的所有测试通过。
lint 干净。
没有 type error。

EACH ITERATION:
1. 运行测试并阅读所有失败信息。
2. 选择影响最大的一个失败点。
3. 写出最小修改来修复它。
4. 重新运行测试、lint 和类型检查。

VERIFY:
测试全绿 + 0 lint warning + 0 type error。

STOP WHEN:
验证通过，或者达到 8 次迭代上限。

ON STOP:
总结改了什么，还有什么仍然失败。

真正的 Loop 底层组成

Claude Code 和 Codex 现在都已经具备这些能力。

1. Automation：让任务自动触发

这是让它变成 Loop 的触发器。

如果你只是手动跑一次，那还不是 Loop。

你需要定义： 提示词 频率 目标

然后让它在你不启动的情况下定时运行，在 Claude Code 里，/loop 可以按间隔重新运行提示词。/goal 会让一个会话持续运行，直到你写下的条件真正满，hooks 可以在 agent 生命周期的不同节点触发命令，cron job 或 GitHub Actions 可以让任务在你合上电脑后继续跑。

结果会回来找你，不再是你四处检查。

2. Skill：可复用的任务说明

不要每次都粘一大段说明，把它保存成一个文件，这个文件会告诉 Loop：

规则是什么 模式是什么 绝对不能碰什么

自动化任务只需要调用这个 Skill，这样，周期性任务才可维护，否则你的工作流会烂在某个没人敢改的定时任务里。

3. Sub-agents：让执行者远离检查者

Loop 里最有用的结构技巧之一，就是把"做事的 agent"和"检查的 agent"分开。

写代码的模型，往往会对自己的结果太友好，另一个 agent，带着不同指令，甚至用更强模型和更高推理强度，能发现第一个模型说服自己忽略的问题。

你的 writer 可以快一点、便宜一点。 你的 reviewer 可以慢一点、严格一点。

这种分离，往往就是质量提升的关键。

4. Connectors：让 AI 真的行动

连接器决定了 agent 是停留在建议层面，还是能进入真实环境执行，差别很大。

前者会说：这是修复方法。 后者会做： 创建 PR 关联 ticket 构建通过后通知频道

Connectors 让 Loop 可以在真实环境里行动，而不是只描述如果它能行动会怎么做。

5. Verifier：真正的关卡

测试、类型检查、构建、规则校验，这些才是真正决定 Loop 是否有价值的东西，其他部分都是管道，Verifier 才是让 Loop 变成真实工作系统的核心，把这些东西叠起来，就得到现在大团队正在规模化运行的东西：

大量 agents 同时围绕同一个任务循环执行，几十个，甚至上千个一起跑。

有工程师用这种 Loop，在大约 6 天内完成了一整个代码库从一种编程语言迁移到另一种语言，如果纯手工做，接近一年，这确实改变了严肃软件构建的方式，但演示里经常不说它的代价。

很多人不提的成本

Loop 消耗 token，token 就是钱，问题还不只是每一步都要花钱，真正的问题是成本会复利式增长，每一轮循环，agent 都要重新读取上下文：

目标 代码 上一轮结果 失败原因

这些内容每一轮都会重新送进模型，而且上下文会越跑越大。，一个跑 10 轮的 Loop，不等于 10 次普通 prompt，它是 10 次越来越大的 prompt，如果你还用了 maker + checker 结构，质量可能提升，但账单也会翻倍，因为两个模型都要读任务。

一个粗略成本可能是：

单 agent，中等任务：约 50,000 - 200,000 tokens 上下文每轮重发：每轮增长 并行 agent fleet：成本继续乘上去

真正该看的指标，应该看每个被接受的有效改动成本是多少？如果 Loop 给你 10 个结果，你丢掉 6 个，那你仍然在做人类审核工作，它本来应该帮你省掉这部分，当接受率低于 50%，Loop 通常就开始亏了，Loop 还会安静地失败。

工程师 Geoffrey Huntley 把一种情况叫做：

Ralph Wiggum loop

意思是 agent 过早判断自己完成了任务，它在半成品状态退出，而 Loop 还在继续运行、继续花钱，却没有产出真正结果，没有硬性验证关卡时，Loop 往往不会明显崩溃，它会悄悄计费，所以重型 Loop 更适合有预算、有监控、有护栏的团队：

迭代上限 token 预算 无聊步骤用便宜模型 监控和告警

如果你还没有这些条件，也不用焦虑，核心思想可以用很低成本的方式先跑起来。

Most Importantly

真正有效的搭建顺序

如果你真的要搭一个 Loop，顺序比工具更重要，能在生产环境里稳定跑的 Loop，通常都是这样来的：

1. 先让一次手动运行稳定 2. 把它变成 Skill，保存指令 3. 给 Skill 套上 Loop，加验证关卡和停止条件 4. 最后再放到定时任务里

最危险的做法是跳过前面几步，直接把一个还没手动跑通的任务放进定时任务，这就是很多 Loop 在你睡觉时爆炸的原因。

先证明一次→再固化→最后自动化

你也可以自己搭一个基础 Loop

你不需要 coding agent，也可以先体验 Loop 的感觉，在任何 LLM 里都能做，关键是一次性给模型三样东西：

目标 严格成功标准 自我检查协议

你可以直接复制这个： 


任务:
[准确描述你想要的产物]

成功标准:
- [标准 1]
- [标准 2]
- [标准 3]

循环内容:
每一轮都重复：
1. PLAN
   说明下一步只做什么。
2. DO
   产出或改进结果。
3. VERIFY
   按每个标准给结果打 1-10 分。
   必须诚实。
   明确列出仍然薄弱的地方。
4. DECIDE
   如果每个标准都达到 8 分以上，输出 FINAL 并停止。
   否则输出日志，继续下一轮，优先修复最低分项。

RULES:
- 所有标准没达到 8 分以前，不能说完成。
- 每一轮必须修复上一轮 VERIFY 中最低分的问题。
- 不要问我问题。做合理假设，写出假设，然后继续。

开始。
一直运行到 FINAL。

你会看到模型开始：

起草 评分 找弱点 修改 再次评分 继续改

它不再把第一版看起来差不多的东西直接交给你，这就是一个 Loop，你用一段提示词就搭出来了，但这里也有缺口，你仍然是触发器，是你打开聊天窗口、是你粘贴提示词、是你坐在那里看它迭代，关闭页面，它就没了，它不会每天早上自动运行，不会在邮件到达时触发，不会主动来找你，如果你想让 Loop 真正自动运行，就要进入前面说的重型世界：

工具 托管 代码 验证关卡 账单

这对真正复杂的任务合理，但对 99% 的日常任务来说，已经有更轻的方案。
