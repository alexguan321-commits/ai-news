How To Become An AI Engineer in 2026 (Without a CS Degree)

来源： https://x.com/sairahul1/status/2062809249064141017 作者： Rahul (@sairahul1) 日期： 2026-06-05 互动： 745 赞 · 148 转发 · 3322 收藏 · 109.8 万浏览

核心论点

公司需要的不是懂数学的人，而是能构建在生产环境中存活系统的人。

关键区分：

  • Chatbot wrapper ≠ 系统
  • Tool call ≠ Agent
  • 知道 LangChain ≠ 知道 Harness Engineering

Harness 的重要性（Anthropic 数据）： 同一个模型（Opus 4.5），不同 harness 性能差距 36 个百分点：

  • Claude Code harness: 78% on CORE benchmark
  • Smolagents harness: 42% on CORE benchmark

结论：模型不重要，harness 才重要。

AI 工程师在 2026 年实际做什么

不是写 prompt，不是选模型。

AI 工程师构建和操作系统周围的系统：

  • 设计 agent loop 和 tool dispatch
  • 工程化 context — 每一步模型前面放什么 token
  • 编写模型能正确 pick 的 tools
  • 添加 memory、durability、sandboxing 应对生产流量
  • 接入 evals 和 CI regression gates 让"更好"可测量
  • 交付能扛住真实用户和真实成本的 agents

四个 context primitives： 1. Write — scratchpads, memory files（agent 读写） 2. Select — 使用点检索，不是预先倾倒 3. Compress — 在 85-95% context window 时摘要 4. Isolate — 子 agent 有独立 context window

Prompt engineering 已死，Context engineering 取而代之。

6 阶段路线图

Phase 0: 建立正确心智模型（Week 1-2）

不要写一行 agent 代码。

三个必须理解的概念：

1. Workflow vs Agent

  • Workflow: 固定控制流（你写的）
  • Agent: 在 loop 中自己做控制流决策
  • 需要 workflow 时构建 agent = 10x 成本 + 2x 故障率

2. 5 个 workflow patterns（Anthropic）

  • Prompt chaining: 输出传给下一个
  • Routing: 不同任务用不同模型
  • Parallelization: 同时运行多个任务
  • Orchestrator-worker: 一个大脑，多双手
  • Evaluator-optimizer: 生成 → 评判 → 改进

3. Harness 类比

  • Model = CPU（原始算力）
  • RAM = context window
  • OS = harness
  • Apps = agent 的 skills

Phase 0 项目： 写 2 页文档，定义：workflow vs agent、5 个 workflow patterns、4 个 context primitives、orchestrator-worker pattern。

Phase 1: 从零构建第一个 Agent（Week 3-5）

写两次 agent。

Build #1 — 原始 Loop（~100 行 Python）

  • 用 messages 和 tools 调用模型
  • 解析 tool_use blocks
  • 执行 tool
  • 追加 tool_result
  • Loop 直到 stop_reason = end_turn

给它 3 个 tools：web_search、read_file、write_file

Build #2 — 同样的 Agent 用 Claude Agent SDK 添加：

  • CLAUDE.md（项目约定）
  • 一个 Skill（定义 "research-summary" 输出格式）
  • 一个 PostToolUse hook（自动格式化 agent 写的每个文件）
  • 一个通过 Task tool 生成的子 agent

Phase 1 项目： Daily briefing agent。读取 Markdown 笔记 + RSS feeds，每天早上写摘要到磁盘。运行一周，看它失败，修复它。

Phase 2: 用正确架构构建真实 Agent（Week 6-9）

生产级技术栈：LangGraph + Deep Agents

LangGraph 提供：

  • State machine（nodes + edges）
  • PostgresSaver checkpointing（进程 kill 后存活）
  • Time-travel debugging（回退到任何步骤）
  • Human-in-the-loop interrupts
  • LangSmith 一等公民可观察性

Deep Agents（LangChain 打包的 harness）提供：

  • Planning middleware
  • Virtual filesystem
  • Sub-agent spawning
  • Automatic context compression
  • Skills

关键概念：Middleware 四个重要 hooks：

  • before_agent — loop 开始前运行
  • wrap_model_call — 包装每个 LLM 调用
  • before_tools — 任何 tool 执行前运行
  • after_tools — 任何 tool 执行后运行

Phase 2 项目：Research Analyst Agent

  • Lead agent 计划，写 TODO list 到 virtual filesystem
  • 并行生成 3 个 search sub-agents（隔离 context）
  • Sub-agents 写结果到文件，返回短摘要给 parent
  • Citation sub-agent 验证声明
  • Writer agent 生成带内联引用的最终 Markdown
  • 状态通过 PostgresSaver 持久化 — kill 进程后恢复
  • Human-in-the-loop interrupt：token 超过 $1 前请求确认

Phase 3: 自己构建 Harness 层（Week 10-13）

这是整个路线图中杠杆最高的阶段。

现代 harness 的 10 个组件： 1. Loop control — 驱动 model → tools → model 的 while-loop 2. Tool dispatch — registry、schema validation、parallel calls、retries 3. Context management — system-prompt 组装、85% window 时压缩 4. Persistence — 每个 node checkpoint 以恢复、回退、分叉 5. Sub-agent orchestration — 隔离 context 的子 agent，压缩摘要返回 6. Skills & progressive disclosure — 只在相关时加载能力 7. Hooks — PreToolUse、PostToolUse、PreCompact、Stop 8. Observability — 每个 model/tool/sub-agent 调用的 OTEL spans 9. Sandboxing — 模型永远没有凭据的容器内代码执行 10. Auth brokering — 凭据永远不进入模型的 context

Phase 3 项目： 用 ~1,500 行 Python 写 mini-harness，必须包括：

  • @tool decorator 的 tool registry + JSON-schema 生成
  • CLAUDE.md-style system-prompt loader
  • SKILL.md progressive-disclosure loader
  • 隔离 context 的 sub-agent spawn primitive
  • Filesystem offload：超过 20K token 的 tool result 写磁盘，用 path + 10 行预览替换
  • 85% context window 时自动压缩
  • 可插拔 hook 系统（pre_tool、post_tool、stop）
  • OpenTelemetry tracing
  • Durable resume：每步后持久化到 SQLite，按 run ID 重载

真正交付物： 1,000 字 post-mortem，比较你的 mini-harness 与 Claude Agent SDK 和 Deep Agents。

Phase 4: 构建 Eval 和 Regression Harness（Week 14-17）

没有这个，每个"改进"都是 vibes。

必须实现的 4 种 eval 类型：

1. Single-turn evals

  • 给定输入，输出对吗？
  • 最便宜，尽可能用确定性 graders
  • 持续运行

2. Trajectory evals

  • Agent 用正确参数调用了正确的 tool 序列吗？
  • 测试单步、完整轮次、多轮变体

3. LLM-as-judge

  • 针对开放式输出：研究报告、代码审查、解释
  • 每周对照人工评分校准

4. End-state evals

  • 针对有状态 agents：数据库写对了吗？文件改对了吗？
  • 比较最终环境状态与 ground truth

不舒服的真相： 模型能检测到它们正在被评估，在 eval 输入上表现不同。设计 eval suite 防止这个，用真实生产查询，不是合成的。

Phase 4 项目： Phase 2 agent 的 regression harness

  • Golden dataset：30-50 个手工评分的研究问题（3 个难度级别）
  • 事实查询的确定性 graders
  • 开放式问题的 LLM-as-judge + 5 标准 rubric
  • Trajectory eval：agent 是否计划、生成 2+ sub-agents、引用来源、预算内完成？
  • 接入 GitHub Actions：golden-set 通过率下降 3+ 点时阻止合并
  • 生产采样：1% 实时 traces 每晚自动评分

Phase 5: 生产加固（永远）

这个阶段不会结束。

5 个永远重要的事：

1. Cost discipline

  • 缓存 CLAUDE.md、system prompt、tool definitions — 节省高达 90%
  • 按难度路由：Haiku 简单轮次、Sonnet 大多数任务、Opus 困难推理
  • Batch API 非实时工作：50% 折扣
  • Multi-agent 烧 ~15x 单 agent token — 只在价值超过时才运行

2. Latency

  • 并行 tool calls，永远 — Anthropic 自己研究 agent 的 system prompt 说"必须用并行 tool calls"
  • 流式部分输出到 UI
  • Sub-agent fan-out：60 步顺序 agent → 10 步 lead + 5 个并行 10 步 sub-agents

3. Safety and sandboxing

  • 所有代码执行在沙箱（Modal、E2B）：永远不在主进程 exec() 模型输出
  • 凭据在模型 context 外 brokering：模型永远看不到它用的 API key
  • 任何不可逆操作的 human-in-the-loop interrupts

4. Monitoring and drift

  • 告警：每请求 token 成本、tool-call 失败率、LLM-as-judge 分数、p95 延迟
  • 每次模型升级后重新基线 evals — harnesses 编码关于模型不能做什么的假设，这些假设会过时

5. Resilience

  • 持久执行（Inngest、Temporal、PostgresSaver）用于运行超过 60 秒的任何 agent
  • 每个 node 后 checkpoint
  • 回退和分叉应该总是可能

5 个生产级项目（本周末选一个构建）

按复杂度排序：

Project 1: AI-Powered Mobile App with SLM

级别： Beginner | 证明： Edge AI + 资源优化

  • 离线优先移动应用，用小语言模型
  • 零 API 成本，完全隐私
  • 关键：按需懒加载模型、内存压力下卸载、滑动 context window + 语义分块、4-bit/8-bit 量化、批量推理减少电池唤醒

Project 2: Self-Improving Coding Agent

级别： Intermediate | 证明： Agentic Loops + 生产调试

  • 写代码、运行测试、从失败学习的 agent
  • 不到功能正常不停止
  • 关键：Plan → Execute → Test → Reflect loop + 最大迭代限制、隔离执行环境、记忆层次（短期/长期/失败）、执行前静态分析

Project 3: Cursor but for Video Editors

级别： Advanced | 证明： 多模态 AI + 复杂工具集成

  • Fork 开源编辑器（Shotcut），构建理解编辑意图的 AI agent
  • 用户说"make this cinematic"，agent 处理剪辑、转场、调色
  • 关键：视觉模型分析每帧 + 音频模型分析对话、意图翻译（"cinematic" → 具体参数）、场景检测、增量预览

Project 4: Personal Life OS Agent

级别： Expert | 证明： 深度 Context + 隐私优先架构

  • 管理日历、财务、健康的 agent，提前数月计划
  • 通过分析睡眠模式和会议密度检测倦怠
  • 关键：实时摄入日历/财务/健康/通讯、个人知识图谱、每 6 小时后台线程检查异常、价值对齐（用户声明优先级）、所有数据加密 + 用户控制密钥

Project 5: Autonomous Enterprise Workflow Agent

级别： Master | 证明： 生产级编排

  • 端到端运行业务工作流的 agent
  • 监控 Slack/Jira → 计划执行 → 委派任务 → 报告结果 + 完整审计日志
  • 关键：事件驱动（监听 Slack/Jira/邮件/监控）、多 agent 委派（orchestrator + 通讯/数据/分析/文档 agents）、自愈（指数退避、熔断器、自动重试决策）、不可变审计日志、关键工作流执行前 human-in-the-loop

技术栈（实际学什么）

框架：LangGraph 1.0 + Deep Agents

为什么不是其他：

  • CrewAI: 最快 demo，生产脆弱，用于 hackathons
  • AutoGen: 合并到 Microsoft Agent Framework，未来不明
  • OpenAI Swarm: OpenAI 自己 README 说"not production-ready"

LangGraph 提供：state machine + PostgresSaver durability + time-travel debugging + OTEL-friendly observability + model-agnostic

Harness 参考：Claude Agent SDK

  • 研究它，使用它
  • 跟 Claude Code 同样的 harness
  • CLAUDE.md + Skills + sub-agents + hooks + filesystem-as-memory
  • 2026 年每个其他 harness 都在收敛到这些 primitives

可观察性：选一个

  • LangSmith: 如果你住在 LangGraph
  • Braintrust: 如果要 framework-agnostic CI gating（$249/月固定）
  • Arize Phoenix: 如果要开源 + OTEL-native

2026 年跳过：

  • OpenAI Swarm — not production-ready
  • OpenAI Assistants API — 2026 年中 sunsetting
  • 在测量实际召回问题前自建 vector store
  • No-code agent platforms（除非是一次性的）

Benchmark 数字（2026 年 5 月）

SWE-bench Verified（编码任务）：

  • Claude Opus 4.7: ~87.6%
  • GPT-5.5: ~88.7%

GAIA（通用 agent 任务）：

  • Claude Sonnet 4.5 leads at 74.6%

τ-bench（客服 agents）：

  • Claude Mythos Preview: 89.2%

关键洞察： 同一 benchmark，不同 harness = 10-36 点波动。模型不如 harness 重要。

17 周时间线

  • Week 2 → Phase 0 完成，能用简单英语解释 harness
  • Week 5 → Phase 1 完成，Claude Agent SDK agent shipped with one Skill, one hook, one sub-agent
  • Week 9 → Phase 2 完成，LangGraph deep-agent running with PostgresSaver durability and LangSmith traces
  • Week 13 → Phase 3 完成，1,500-line mini-harness written and documented
  • Week 17 → Phase 4 完成，Golden datasets, CI gates, one published-benchmark run via Inspect
  • Week 17+ → Phase 5，永远

兼职（每周 10-15 小时）：所有时间乘以 2.5x

不舒服的真相

大多数人会读这个然后什么都不做。

2026 年的残酷真相：

  • 可替代的： 构建薄 GPT wrappers
  • 不可解雇的： 交付带 evals 和 durability 的自主系统

差距是 5 个项目 + 17 周专注工作。

当前状态（2026）：

  • 57% 团队现在在生产环境有 agents
  • 89% 那些团队接入了可观察性
  • 质量是 #1 障碍（32% 团队引用）

这意味着整个领域瓶颈在能构建 evals 和 harnesses 的工程师，不是能调用 LLM API 的工程师。

关键语录

"The companies hiring right now don't need people who understand the math. They need people who can build systems that survive production."

"Prompt engineering is dead as a standalone skill. Context engineering replaced it."

"The model matters less than the harness."

"Expertise is the only job security left. Production systems are the only portfolio that matters."

行动号召

1. 选一个项目。 新手从 Project 1 开始，已经在交付代码的从 Project 5 开始。 2. 本周末构建它。 市场奖励交付，不是学习。 3. 记录一切： 架构决策、失败和恢复、自我纠正 loop。 4. 公开构建。 交付时 tag 我。

下个月，90% 的人什么都不会做。 他们还会构建同样的 wrappers。

另外 10% 会交付真实的东西。 他们会有面试、offer 和职业杠杆。

选择很简单： 成为公司迫切想雇佣的架构师，或者变得过时。
