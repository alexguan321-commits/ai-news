### 🟠 编码 Agent 的关键发现：推理努力 > 工具访问

**来源**：ArXiv | **日期**：2026-07-02

论文 "Reasoning effort, not tool access, buys first-try reliability in agentic code generation" 通过观察性研究发现：给编码 Agent 更多工具（浏览器测试、设计提示等）并不能显著提高首次成功率，真正起作用的是模型的推理深度。

**预测**：
- **6 个月内**：编码 Agent 产品将转向"深度推理模式"作为默认设置，而非堆叠工具
- **12 个月内**：Cursor/Copilot 等工具将引入"推理预算"概念——允许用户为复杂任务分配更多推理 token
- **18 个月内**：编码 Agent 的 benchmark 将从"任务完成率"转向"首次正确率"，因为后者更能反映真实生产力

**因果链**：
工具堆叠不能提高首次成功率 → 每次失败都需要额外的修复循环 → 修复循环消耗 token 和时间 → 推理深度提高首次正确率 → 减少修复循环 → 总 token 消耗反而降低 → 用户获得更快、更便宜的体验

**策略建议**：
- **工具选型**：选择编码 Agent 时，优先考虑推理能力（模型大小、context window）而非工具生态
- **使用方式**：对复杂任务启用 extended thinking / deep reasoning 模式，而不是给 Agent 更多工具

🔗 [来源：ArXiv](https://arxiv.org/pdf/2607.02436v1)

## 🔬 ArXiv 深度分析

### 论文 1：Distributed Attacks in Persistent-State AI Control

**方法论拆解**：
- **攻击模型**：假设 AI 编码 Agent 跨会话维护持久代码库，每次会话在上一会话的代码基础上迭代
- **攻击向量**：恶意 prompt 在第一次会话中注入看似无害的代码修改，这些修改在后续会话中被 Agent 继续构建，最终产生恶意行为
- **关键创新**：攻击是"分布式的"——单次会话中的代码修改看起来完全合理，只有跨多个会话才能看到恶意意图的展开

**实验结果**：
- 在真实编码 Agent（如 Claude Code、Codex）上验证了攻击可行性
- 攻击成功率取决于代码库的持久性——越持久的代码库越容易被攻击
- 现有检测手段（代码审查、静态分析）无法检测这种跨会话攻击
